Mira Onboarding — Prototype

PM test assignment · goal: maximize D1 retention · design is secondary — idea & flow first

The idea in one line

Value in 60 seconds → visible memory → a next-day hook set in one tap. Onboarding doesn't pitch features — it does something useful now and makes a deal about tomorrow.

Production today vs this prototype

Production today/start → 6-screen mini-app form (role, intent, name) → skill suggestions → setup via 3 open-ended questions. Zero value delivered in the first session. Abandon the setup → Daily never arrives; the recovery push is generic and, again, button-less.
This prototypeValue in the first 60 seconds (Mira acts first) → memory is visible and user-controlled → Daily set in one tap, every question optional → the morning push degrades honestly (tiers A/B/C), one CTA per message.

How to review (~2 min)

  1. Pick a track — or type anything: free text never gets blocked. That's a principle.
  2. Follow to the end — one-tap hook setup, optional follow-ups.
  3. Hit "Skip to tomorrow" — the D1 morning push arrives.
  4. Switch push tiers A / B / C — how the push honestly degrades with less data.
  5. ↺ in the header restarts — 4 tracks plus an opt-out branch.

Assumptions (no data from the team yet)

  • 60 🪙 welcome grant in the Content track: first image 30, optional re-roll 30 (new users currently get 0 — paywall shock instead of a wow). COGS cap ≈ two top-model images per Content newcomer.
  • First generation and the first push run on the top model — a Pro teaser at a fixed, tiny COGS. Daily text briefs stay free on a budget model.
  • Default push time 8:30; time zone estimated from the /start hour. Pushes auto-pause after 3 unanswered mornings (stop rule).
Based on PRD v0.8: production onboarding audit, 5-competitor benchmark, live field tests, two internal review rounds. All data in the prototype is demo.
